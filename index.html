Part 1 · The area
02 · Feature choice
The chosen feature area

I'd start with Add to CRM.

Three reasons why, and three ways I'd stress-test that choice before building anything.

Three reasons why
  1. 1
    It is the activation metric.
    Surfe defines activation as 3 meaningful actions on day 1, and those actions are primarily Add to CRM.
  2. 2
    It's exactly where users stall.
    The blank slate lives between setup and this first action. Fix the path to this event, fix the 30% do-nothing.
  3. 3
    Shortest path from signup to retention.
    Enrichment, recommendations, integrations all sit downstream. Add to CRM is the gate.
Three ways I'd stress-test it
Talk to
  • CS & the do-nothing cohort, microsurveyed in-extension
  • Recently-activated SDRs: what clicked between install and first Add to CRM?
  • External SDRs & AEs not on Surfe: walk them through the funnel cold
Look at
  • Time-to-first-Add-to-CRM, cut by channel, role, CRM choice
  • Session replay on 50 post-install web app + LinkedIn sessions
  • Day-1 activators vs non-activators: retention curve delta
Watch for
  • "Didn't know what to do" > "couldn't connect" in interviews
  • Dwell-then-leave post-install, consistent across CRMs (rules out setup)
  • Support tickets on CRM connection would refute it, point back to setup
If signals point elsewhere, I pivot: wrong-fit users → targeting, CRM connection pain → integrations.
Part 1 · Opportunity
03 · Opportunity
Why this is worth doing first

One intervention. Three metrics move.

The problem
Setup is easy. The workflow is invisible. 30% of installers never take an action, 75% of signups never experience Surfe's value.
My proposal
Replace the blank slate after setup with a guided first prospecting session that lands new users on their first Add to CRM, inside LinkedIn, not on an empty dashboard.
Activation

Halve the do-nothing rate.

Users arrive already in the workflow, not staring at a blank screen.

+2K activated users / year
Retention · MAU

First session teaches the habit.

Users who learn the daily workflow on day 1 are more likely to repeat it in week 2, month 2, month 6.

40% → 60% MAU% target
ARR

Activation lifts volume. Retention compounds it.

~€35–170K from activation alone. Every 1pp off monthly churn is worth multiples more.

2× ARR if churn halves
Retention is the bigger lever. Order of magnitude estimates, directional not precise.
Part 2 · Feature design
04 · Today
Today's onboarding

Setup before value, features before workflow.

Three things the current flow asks of a new user before they see anything worth coming back for.

Issue 1
Blank slate after setup.
Right after CRM connect, the home page is empty except for a CTA to set up a "Co-pilot." No prospects, no workflow, no obvious next click.
Empty state, no prospects
Issue 2
Target customer setup gate.
Users have to define who they're looking for (size, role, industry) in the abstract, before seeing any real prospect or feeling a single moment of value.
Abstract filters, no preview
Issue 3
Tour teaches features, not workflow.
Tooltips point at filters and UI affordances. Nothing shows what a day of prospecting with Surfe actually looks like, or bridges to the extension.
"Let's take a look around"
My read
Onboarding is setup-first and web-app-only. The extension (the actual daily habit surface) isn't part of the story.

Part 2 · Scope
06 · Scope
MVP scope

Focus on habit forming.

v1 · Ships first
The populated dashboard + the first sync.
~2 weeks, small empowered team
  • Target quiz at the end of onboarding (2 questions)
  • Dashboard populated with ICP-matching companies on arrival
  • Famous-tech fallback for skippers
  • Hover-sync on contact cards (existing Surfe pattern)
  • "X of 3 today" progress tracker
  • LinkedIn sidebar nudge on the dashboard
  • Event instrumentation across the new funnel
v2 · After v1 learnings
Deepen the habit.
Scope decided after first A/B reads
  • Seed logic tuning from sync / dismiss signals
  • Re-engagement nudges for day 2, day 7 drop-offs
  • Cross-surface progress sync (dashboard ↔ extension)
  • Richer target refinement inside the dashboard
Keep the scope clean This is one hypothesis test. Bundling extension UX, enrichment changes, pricing tweaks or integration work muddies what we learn from the A/B.
Part 3 · Execution
07 · Execution
How we'd ship it

2 weeks, synchronously, one small team.

What we build, in parallel
PM + Design ~80% dedicated designer
Co-design the target quiz + dashboard arrival. Reuse existing Wave components, no new CSS surface.
Engineering 2 FE + 1 BE
ICP match query, quiz capture, dashboard state, event layer. Ship a feature-flagged slice every few days.
Data + QA in parallel
Stand up session replay + funnel dashboards on day 1. Same tools we'll read the A/B with.
Bringing them
along the journey
Design Paired from day 0. I bring problems, they own the solution. Reviews every 2 days, design leads the visual-system calls.
Engineering Unknowns doc owned jointly from week 0. Tradeoffs decided in the open. Eng leads arch; I back them and unblock. Demo-every-Friday.
Support / CS Dogfood the beta in sprint 1. Co-write the FAQ + canned replies before launch. Weekly readout with CS leads the first month post-launch.
Unknowns to validate first
Week 0, before any feature code.
  • Is the companies/contacts endpoint fast enough to load 3 ICP-matching rows on dashboard mount?
  • Can the extension reliably access recent LinkedIn profile views across session states? (seed fallback covers us if not)
  • What's the webhook path from extension sync back to the dashboard's progress state?
  • Tour tooltips from today's onboarding will collide. Do we remove them cleanly, or A/B the transition?
  • Target quiz answer model: fixed roles/industries (fast) or freeform (accurate)? Need eng + design call.
Part 4 · Measurement
08 · Measurement
Measuring success

One primary metric. Many to watch. Strict test length.

Primary metric
Day-1 activation rate 3 "Add to CRM" actions within 24h of signup, from ~25% baseline
+5% rel.
Test design: 50/50 A/B across the entire funnel, all new signups. MDE 5% relative on day-1 activation at 80% power, ~19K users per arm.

Length is set by the slowest metric: activation reads in ~4-6 weeks, paid conversion needs 12-16 weeks minimum. We commit to the full length before starting, no early calls.
Counter & guardrail metrics
Retention curve day 7, week 2, week 4. Depth of the habit, not just the spike.
watch
Trial-to-paid conversion slowest, noisiest, most consequential lagging metric
watch
Early cancellations (< 30 days) activating low-intent users would drive this up
counter
Signup → onboarded conversion did the new quiz cost us anyone at the top?
counter
Usage depth credits consumed + outreach on synced contacts. Are reps actually prospecting, or hoarding?
watch
Support ticket volume unexpected confusion during the transition
watch
Phase 1 Build visibility ~3 weeks · Instrumentation, dashboards, session replay, experimentation tooling stood up
Phase 2 50/50 A/B, full funnel 12-16 weeks min · Strict, don't stop on activation signal. Paid conversion drives length.
Phase 3 Soft launch, A/B preserved If activation signal is clear: new experience for 100% of NEW signups. A/B cohort stays intact for lagging reads.
Ongoing Iterate on what we learn Signal-driven, not scheduled. Surprises on primary or guardrail metrics trigger a focused iteration, same test rigor as v1.

Team invite loops.

Paid channel quality audit.

Targets: reactivation rate, cheaper ARR per $
A parallel bet
10 · A bolder bet
Worth exploring in parallel, this year

Surfe's next user
might not be a human.

AI agents already prospect, enrich, and write to CRMs in CLI and IDE environments. They're a new buyer, a new surface, and a new workflow. The Surfe data moat is just as useful to them, if we meet them where they work.

Separate team, separate product shape. Not a roadmap footnote, a real bet.

New surface · CLI + MCP Agents live in terminals, IDEs, Claude Code-style environments. They don't browse; they call tools. Surfe ships an MCP server and a proper CLI alongside the web API.
New buyer · AI dev ecosystem Not only end reps. Agent framework builders, AI sales tools (Cursor-for-sales, vertical agents). Different sales motion, different pricing, same data moat.
Moat stays human Maged's right that humans still close deals. Surfe powers whatever finds them, human UI or agent tool call. The valuable layer is the data, not the screen.
